# Repository Guidelines

## Project Structure & Module Organization
- `src/contents/` contains three page-specific content scripts:
  - `page1.tsx`: dashboard panel UI, run control, queue scheduling, and page3 capture display.
  - `page2.tsx`: auto-click flow, account request interception, open page3.
  - `page3.tsx`: hook + auto-click + request capture + completion reporting.
- `src/background.ts` is the runtime coordinator; state persistence, message routing, tab operations, and progress updates.
- `src/core/` contains shared protocol and logging abstractions:
  - `core/protocol/messages.ts`: all runtime message constants/types.
  - `core/logging/logger.ts`: unified runtime logger.
- `src/domain/` contains business parsing and queue logic:
  - `domain/capture/parsers.ts`: `shop_id`/`buyinId` extraction and capture grouping.
  - `domain/run/queue-controller.ts`: queue/cache/dedupe controller.
- `src/services/` contains infrastructure wrappers:
  - `services/runtime/messenger.ts`: safe `chrome.runtime.sendMessage` wrapper.
  - `services/injection/script-injector.ts`: fetch-hook injector helper.
- `src/lib/` retains low-level browser utilities (`inject-script.ts`, `simulate-click.ts`).
- `assets/fetch-hook.js` is the injected hook script for fetch/xhr interception.
- `build/` is Plasmo output generated by `pnpm build`/`pnpm dev`.

## Current Workflow Summary
- Page1 intercepts `/selection/common/material_list`, parses `data.summary_promotions[].base_model.shop_info.shop_id`, and opens page2 with `shop_id` parameter.
- Page2 auto-clicks target button, intercepts `/connection/pc/im/account`, parses `data.buyin_account_id`, then opens page3 with `buyinId` parameter.
- Page3 captures `/connection/pc/im/shop/contact` and `/connection/pc/im/shop/detail`, forwards payloads to page1 for grouped JSON display (`contact`/`detail` latest only), and reports completion.
- Page2/Page3 close themselves 3 seconds after completion triggers.
- New tabs open in background by default (`active: false`) to keep page1 focused.

## Runtime Message Protocol (Canonical)
- State:
  - `state/get`
  - `state/update`
  - `run/start`
  - `run/stop`
  - `progress/current`
  - `progress/completed`
- Tab actions:
  - `open/url` (canonical)
  - `page1/open-page2` (legacy compatibility)
  - `tab/close-self`
- Data/log:
  - `page3/capture`
  - `log`

## Build, Test, and Development Commands
- `pnpm dev`: start the Plasmo dev server and build a live-reload extension bundle for local testing.
- `pnpm build`: create a production build for store submission.
- `pnpm package`: package the production build into a distributable archive.

## Coding Style & Naming Conventions
- TypeScript + React with functional components; JSX files use `.tsx`.
- Indentation is 2 spaces and semicolons are omitted (match existing style).
- Component names use `PascalCase` (e.g., `CountButton`), file names are `kebab-case` (e.g., `count-button.tsx`).
- Prettier is available with `@ianvs/prettier-plugin-sort-imports`; keep imports grouped and sorted.
- Keep message type strings centralized in `core/protocol/messages.ts`; do not add ad-hoc strings in page files.
- Prefer moving parsing, queue, and state transition logic into `domain/*` modules; keep page entries thin.
- Add concise Chinese comments only at critical logic boundaries (protocol contracts, completion conditions, queue/state transitions).

## Testing Guidelines
- No test framework is currently configured. If you add tests, document the runner and add a `test` script in `package.json`.
- Prefer colocating tests near features (e.g., `src/features/__tests__/feature.test.tsx`) and use clear, behavior-driven names.

## Commit & Pull Request Guidelines
- Git history is minimal and does not establish a commit message convention; use concise, imperative messages (e.g., "Add options page").
- Pull requests should include a short summary, testing steps (manual or automated), and screenshots or screen recordings for UI changes.

## Security & Configuration Tips
- Extension permissions live in the `manifest` field within `package.json`; keep host permissions as narrow as possible.
- When adding new pages (options, content, background), follow Plasmo conventions so builds pick them up automatically.
